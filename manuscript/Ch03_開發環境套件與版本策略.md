## 3.1 認識 uv：更快的 Python 套件與專案管理工具

本書的安裝指令一律使用 **uv**。在用它之前，先花一節把它講清楚——畢竟推薦一個你沒介紹過的工具，對讀者並不公平。

**uv 是什麼？** 它是一個用 Rust 寫的、**速度極快的 Python 套件與專案管理工具**，出自 Astral（也就是做出知名 linter `ruff` 的團隊）。你可以把它理解成「一個工具，取代過去一整串工具」：以往你可能要同時用 `pip`（裝套件）、`venv` / `virtualenv`（建虛擬環境）、`pip-tools`（鎖版本），uv 把這些功能整併成單一指令集。

**為什麼本書選它？** 兩個對我們特別重要的理由：

第一是**速度**。uv 的相依解析與安裝比傳統 pip 快上一個數量級（官方宣稱在多數情境快 10–100 倍）。當你跟著本書反覆建環境、試不同 provider 套件時，這個差異很有感。

第二，也是更關鍵的——**可重現性**。uv 會自動產生並維護一個 `uv.lock` 檔，精確記錄整棵相依樹的每一個版本。這直接呼應本章稍後（3.7 節）要強調的「版本鎖定」原則：在 LangChain 這種改版頻繁的生態裡，一個能保證「今天能跑、三個月後重建環境還能跑」的 lockfile，是不可妥協的基本功。

**它和 pip 有什麼不同？** 一句話：**pip 只管「把套件裝進你自己管理的環境」，uv 則同時管理「專案 + 虛擬環境 + lockfile」三件事。** 最常見的對照是：

| 你想做的事 | pip（+venv）| uv |
|---|---|---|
| 建專案 | 手動建資料夾、`python -m venv .venv` | `uv init` |
| 加一個套件 | `pip install langchain` | `uv add langchain` |
| 移除套件 | `pip uninstall langchain` | `uv remove langchain` |
| 依 lockfile 還原環境 | `pip install -r requirements.txt` | `uv sync` |
| 在環境裡跑指令 | 先 `activate` 再跑 | `uv run python agent.py` |

**安裝 uv 本身**（只需做一次）：

```bash
# macOS / Linux（官方安裝腳本）
curl -LsSf https://astral.sh/uv/install.sh | sh

# macOS 也可用 Homebrew
brew install uv

# 任何平台：若你已有 pip，也能直接裝
pip install uv

# Windows（PowerShell）
# powershell -c "irm https://astral.sh/uv/install.ps1 | iex"
```

裝好後確認：

```bash
uv --version
```

**常用指令速查**（本書會反覆出現）：

```bash
uv init                  # 在當前資料夾初始化一個新專案（建立 pyproject.toml）
uv add <套件>            # 加入相依並安裝，同時更新 pyproject.toml 與 uv.lock
uv remove <套件>         # 移除相依
uv sync                  # 依 pyproject.toml / uv.lock 還原出一致的環境
uv run <指令>            # 在專案環境中執行指令（不必手動 activate）
uv lock                  # 重新解析並更新 uv.lock
uv pip install <套件>    # 相容層：當你只想要「pip 的行為」時可用
```

最後一行值得一提：uv 提供一個 `uv pip ...` 的**相容層**，行為幾乎等同 pip。所以即使你習慣 pip 的思維，也能在 uv 底下無痛操作。

**如果你就是想用 pip 呢？** 完全沒問題。本書安裝一律以 uv 示範；偏好 pip 的話，對照上表把 `uv add X` 換成 `pip install X` 即可，書中的 agent 程式碼一行都不會因此改變。uv 是為了讓你更省事，不是硬性規定。

## 3.2 安裝核心套件

有了 uv（或 pip），來裝本書的三個核心套件。它們由下而上對應到 Ch2 的三層：`langgraph`（runtime）、`langchain`（framework）、`deepagents`（harness）。先確認 Python 版本：

```bash
python3 --version   # 需要 3.10 以上
```

安裝：

```bash
uv init                                  # 初始化專案（建立 pyproject.toml）
uv add langchain deepagents langgraph    # 加入相依並安裝
uv sync                                  # 依 lockfile 還原一致環境
```

> 小提醒：裝了 `langchain` 與 `deepagents` 後，`langgraph` 通常會被當作相依套件一起裝進來（因為兩者都蓋在它上面）。我們仍然明確列出 `langgraph`，因為 Part I 會直接、大量地使用它。

**Provider 套件要另外裝。** LangChain 本身不綁任何特定模型廠商——它提供上百個整合，但這些整合住在各自獨立的套件裡。你用哪家模型，就裝哪家的套件。本書以 Google Gemini 為主：

```bash
# Google Gemini（本書主要使用）
uv add langchain-google-genai

# 其他常見 provider（按需選裝）
uv add langchain-openai             # OpenAI
uv add langchain-anthropic          # Anthropic / Claude
# fireworks、bedrock… 各有對應套件，見官方 integrations 頁
```

## 3.3 版本地雷：`langchain` 1.0 vs `langchain-classic`

這是新讀者最容易踩、且最容易莫名其妙的坑，先講清楚。

LangChain 在 **2025-10** 釋出了 **1.0**，做了一次大改版：**把舊有那一大堆 chains 與 agents 全部移除，只留下唯一一個高階抽象——蓋在 LangGraph 上的 `create_agent`。** 這是個刻意的「瘦身」：過去 LangChain 塞了太多高階便利介面，反而讓人困惑、也難以維護；1.0 把這些砍掉，回歸「標準化模型 I/O + 一個乾淨的 agent 抽象」。

這帶來一個實際後果：**網路上、舊書裡、舊教學影片中大量的 LangChain 程式碼，在 1.0 跑不動。** 那些 `from langchain.chains import ...`、`initialize_agent(...)`、`AgentExecutor` 之類的寫法都屬於舊世界。

官方給的退路是：如果你有非搬不可的舊程式、暫時不想升級，可以改裝 **`langchain-classic`** 套件繼續用舊 API（但官方建議你還是遷移）。

**本書的立場很明確：我們只教 1.0 的世界**（`create_agent`、middleware、蓋在 LangGraph 上的現代寫法）。當你在外面看到對不上的舊範例，先確認它是不是 0.x 時代的產物。第 52 章會專門談「從舊 LangChain 遷移」。

> ⏱ **三秒判斷** —— Python 端跑 LangChain 報錯，先看錯誤類型就能猜八成：
> - `ImportError: cannot import name 'create_agent'` → **裝到 0.x**，升到 1.x
> - `ModuleNotFoundError: No module named 'langchain_xxx'` → **provider 套件沒裝**（`uv add langchain-xxx`）
> - `AttributeError: ... has no attribute 'with_structured_output'` → **碰到 0.x 舊 API**，遷移或暫裝 `langchain-classic`
> - `KeyError: 'GOOGLE_API_KEY'`／`ANTHROPIC_API_KEY` → **金鑰沒設或拼錯**

判斷你裝到哪個版本：

```python
import langchain
print(langchain.__version__)   # 應該是 1.x.x
```

## 3.4 設定 API key

LangChain 透過環境變數讀取各家 provider 的金鑰。本書以 Google Gemini 為例：

```bash
export GOOGLE_API_KEY="你的-api-key"
```

其他 provider 同理，換成對應的變數名（節錄自官方支援清單）：

```bash
export OPENAI_API_KEY="..."         # OpenAI
export ANTHROPIC_API_KEY="..."      # Anthropic / Claude
export OPENROUTER_API_KEY="..."     # OpenRouter
# AWS Bedrock 需要 AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY / AWS_REGION
```

> 實務建議：別把金鑰寫死在程式碼裡。開發時用 `.env` 檔搭配 `python-dotenv`，並把 `.env` 加進 `.gitignore`；上 production 則交給機密管理服務。第 50 章談安全時會再細講。

## 3.5 跑通你的第一個 agent

環境就緒，我們跑一個最小但完整的 agent。它用一個（假的）天氣工具，展示 `create_agent` 如何一行包好 Ch1 那整個 agent loop：

```python
from langchain.agents import create_agent

def get_weather(city: str) -> str:
    """查詢指定城市的天氣。"""
    return f"{city} 一直都是晴天！"

agent = create_agent(
    model="google_genai:gemini-2.5-flash",
    tools=[get_weather],
    system_prompt="你是個樂於助人的助理。",
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "舊金山天氣如何？"}]}
)
print(result["messages"][-1].content_blocks)
```

把它存成 `agent.py`，用 `uv run python agent.py` 執行。如果你看到一段提到舊金山天氣的回覆，恭喜——你的環境通了。

這裡有兩個寫法值得記住，後面會反覆出現：

- **`model="google_genai:gemini-2.5-flash"`**：模型用字串指定。也可以用 `"provider:model"` 的格式，例如 `"openai:gpt-5.4"` 或 `"google_genai:gemini-2.5-flash-lite"`，方便跨 provider 切換（第 20 章詳談）。
- **`{"messages": [...]}`**：agent 的輸入是一個帶 `messages` 的 dict；輸出也在 `result["messages"]` 裡，最後一則就是 agent 的回覆。`.content_blocks` 是 1.0 的標準化內容格式，能容納文字、reasoning、引用等多種區塊。

## 3.6 開啟 LangSmith tracing

agent 是個「會自己決定走幾步」的東西，當它行為怪異，你最需要的是**看見它每一步到底做了什麼**。LangSmith 就是做這件事的可觀測性平台，而且開啟它幾乎零成本——設三個環境變數就好：

```bash
export LANGSMITH_TRACING="true"
export LANGSMITH_API_KEY="你的-langsmith-key"
export LANGSMITH_PROJECT="langgraph-deepagents-book"   # 選填，用來歸類專案
```

設好後，**完全不用改你的程式碼**——重新跑一次 3.5 的 agent，到 LangSmith 網站上就能看到這一次執行的完整 trace：模型收到什麼、決定呼叫哪個工具、工具回傳什麼、最後怎麼收尾。

本書強烈建議你**從第一天就把 tracing 開著**。它不只是上線後才用的工具；在學習階段，能「看見」agent 的每一步，是建立正確直覺最快的方式。第 45 章我們會深入 tracing 的進階用法。

## 3.7 專案結構與版本鎖定策略

最後談一件容易被忽略、但會在三個月後反咬你的事：**版本鎖定**。

LangChain 生態改版很快（你已經在 3.3 見識過 1.0 的大改）。如果你的相依只寫 `langchain`（不指定版本），那麼今天能跑的程式，下次重裝環境可能就因為某個套件升級而壞掉。對一本要長期維護的書、或一個要長期運作的 production 專案，這是不可接受的。

策略很簡單：**把版本鎖死，並把鎖定檔納入版控。** 這也正是 3.1 介紹 uv 時強調的那個賣點——`uv.lock` 會自動幫你做到這件事。

```toml
# pyproject.toml（uv 會幫你維護這個檔）
[project]
dependencies = [
    "langchain==1.0.3",
    "langgraph==0.x.y",
    "deepagents==0.5.z",
    "langchain-google-genai==0.x.y",
]
```

```text
# 若你用 pip，可改用 requirements.txt
langchain==1.0.3
langgraph==0.x.y
deepagents==0.5.z
langchain-google-genai==0.x.y
```

uv 會自動產生 `uv.lock`（完整相依樹）；pip 則可用 `pip freeze > requirements.lock` 凍結。**把這些 lock 檔 commit 進 git**，任何人（包括三個月後的你）都能用 `uv sync`（或 `pip install -r`）重建出一模一樣、保證能跑的環境。

> 🩸 **血淚教訓** —— 我曾經接手一個半年前的 agent 專案，README 寫「`pip install langchain==0.0.350`」。照辦後 `langchain-openai` 解析到新版、`langchain-core` 跟舊 langchain 不相容，連 import 都炸。**沒 lock 檔，相依樹只是「看起來鎖定」，每次重裝其實都是一次新的解析**。從那天起我有條鐵則：**任何上 production 的 agent，pyproject.toml + `uv.lock`（或 `pip freeze` 出的 lock）都 commit 進版控**，少一個都不行。

一個推薦的最小專案骨架：

```
my-agent/
├── pyproject.toml        # 相依與版本鎖定（uv 維護）
├── uv.lock               # 完整相依樹（commit 進版控）
├── .env                  # API keys（加進 .gitignore，別 commit）
├── .gitignore
└── src/
    └── my_agent/
        ├── __init__.py
        ├── agent.py      # agent 定義
        └── tools.py      # 工具實作
```

本書每一章開頭都會標明使用的套件版本，正是這個原則的體現。你跟著做時，建議也養成「先鎖版本、再寫程式」的習慣。

## 常見坑（Pitfalls）

- **照抄網路上的舊 LangChain 範例卻一直報錯**：八成是 0.x 時代的程式（`AgentExecutor`、`initialize_agent`、各種 `chains`）。先確認版本，本書只走 1.0。
- **在 uv 專案裡又手動 pip 裝東西**：這會讓 `uv.lock` 與實際環境不同步、破壞可重現性。在 uv 專案裡請一律用 `uv add`，真的要用 pip 行為時走 `uv pip`。
- **不鎖版本**：「在我電腦上明明能跑」最常見的元兇。沒有 lock 檔，環境就不可重現。
- **忘了裝 provider 套件**：只裝 `langchain` 卻沒裝 `langchain-google-genai`，呼叫 Claude 時會報找不到整合。模型廠商的套件要分開裝。
- **把 API key 寫進程式碼並 commit**：金鑰外洩的頭號途徑。用環境變數或 `.env`，並確保 `.env` 在 `.gitignore` 裡。
- **遇到 agent 行為怪異卻沒開 tracing**：等於蒙著眼除錯。設三個環境變數就能看見全貌，沒有理由不開。

## 本章小結

我們先認識了 **uv**——一個快速、整合套件／虛擬環境／lockfile 的 Python 工具，它的 `uv.lock` 正好支撐本書最在意的「可重現環境」；不愛 uv 的人，全書每個步驟也都有 pip 版本可循。接著我們裝好 `langgraph` / `langchain` / `deepagents` 三件套與 provider 套件（Python 3.10+），釐清了 **`langchain` 1.0 砍掉舊 chains/agents、只留 `create_agent`** 這個關鍵版本斷層（舊程式可暫用 `langchain-classic`，但本書只教 1.0）。我們設好 API key、跑通了第一個 `create_agent`，並用三個環境變數開啟了 LangSmith tracing——從今以後你都能「看見」agent 的每一步。最後訂下版本鎖定策略：把版本鎖死、把 lock 檔納入版控。環境到位了，下一章起我們正式潛入最底層的 LangGraph，開始親手把 agent loop「畫」出來。

## 延伸閱讀 / 練習

1. **建立你的書用環境**：用 `uv init` 照 3.7 的骨架建一個專案，`uv add` 鎖好版本、commit `uv.lock`，後續每章的程式都寫在這裡。（偏好 pip 的話，用 venv + `requirements.txt` 等效完成。）
2. **uv vs pip 體感**：分別用 `uv add` 和 `pip install` 裝一次 `langchain`，感受一下安裝速度差異。
3. **換模型跑跑看**：把 3.5 的 `model` 換成你有金鑰的另一家 provider（記得先裝對應套件），確認跨 provider 切換真的只是改一個字串。
4. **讀你自己的 trace**：開啟 tracing 後重跑 3.5，到 LangSmith 上找出這次執行裡「模型決定呼叫 `get_weather`」的那一步，看看它帶了什麼參數。
5. **版本偵錯演練**：故意把 `langchain` 降到某個 0.x 版本，跑 3.5 看它怎麼壞，再升回 1.x。親身體會一次版本斷層，你會記得更牢。
