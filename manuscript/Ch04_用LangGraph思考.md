# Ch 4　用 LangGraph 思考：node、edge、state 三件事

> **本章目標**
>
> - 建立 LangGraph 的核心世界觀：把一個 agent／工作流，想成一張由 **node、edge、state** 組成的圖。

## 4.1 為什麼是「圖」？

Ch1 我們刻了一個 agent loop，Ch2 看到 LangChain 的 `create_agent` 一行就把這個 loop 包好。那為什麼還要學底層的 LangGraph？

因為真實任務很少是「一直 loop 到結束」這麼單純。它們有**分支**（這封信是客訴還是詢價，要走不同處理）、有**平行**（同時查三個資料源）、有**「某些步驟要照固定順序、某些步驟才交給模型自由發揮」的混合**。`create_agent` 那個固定的 loop 包不住這些形狀，這時你就需要能**精確描述流程結構**的工具——這正是 LangGraph 的核心。

LangGraph 的世界觀只有三個元件，請牢牢記住，整個 Part I 都在它們之上展開：

- **State（狀態）**：一個共享的資料結構，代表你的應用「此刻的快照」。它是所有節點共讀共寫的「白板」。
- **Nodes（節點）**：一個個 Python 函數，封裝 agent 的邏輯。它讀入當前 state，做一些運算或副作用，回傳對 state 的更新。**節點負責「做事」**——裡面可以是一次 LLM 呼叫，也可以只是普通程式碼。
- **Edges（邊）**：決定「下一個跑哪個節點」。可以是固定的轉移，也可以是依 state 而定的條件分支。**邊負責「決定往哪走」**。

一句話總結：**節點做事，邊決定接下來做什麼。** 把節點與邊組合起來，你就能描述出會分支、會迴圈、會隨狀態演化的複雜工作流。

> 💡 **冷知識** —— 「state + 一堆改變 state 的函數 + 決定下一個函數的規則」這個世界觀在電腦科學裡有個老名字叫 **state machine（狀態機）**：從 1940 年代有限自動機、1990 年代 UML statechart，到 Erlang 的 actor、React 的 `useReducer`，骨架本質都是這個。LangGraph 的新意，是讓「自動機的每一步可以是一次 LLM 推理」——但結構還是那個從 1940 年代沿用至今的老朋友。

## 4.2 一個真實任務：客服 email agent

抽象講不如拿真東西練。本書 Part I 會反覆用同一個情境——**客服 email agent**——來示範各種機制。它的需求是：

> 收到客戶來信後，agent 要：讀信 → 依緊急度與主題分類 → 查文件找答案 → 草擬回覆 → 複雜案件升級給真人 → 必要時安排後續追蹤。

要處理的信千奇百怪：「我要怎麼重設密碼？」（簡單問題）、「匯出 PDF 會當掉」（bug 回報）、「我的訂閱被收兩次錢！」（緊急帳務）、「能不能加個深色模式」（功能許願）、「我們的 API 整合會間歇性回 504」（複雜技術問題）。

這個情境之所以好，是因為它同時踩中 Ch1 講的三個限制：需要外部資訊（查文件、查客戶資料）、需要實際動作（建 bug 單、送信）、而且後面的步驟取決於前面的結果（怎麼回覆取決於分類）。它逼著我們用「圖」來思考。

## 4.3 五步思考法

每次要用 LangGraph 實作一個流程，都可以照下面五步走。這套方法本身比任何單一 API 都值得記住。

**第一步：把流程畫成一個個離散步驟（→ 這些會變成節點）**

先別碰程式，拿張紙，把流程拆成一個個「只做一件事」的步驟，再畫出它們怎麼連接：

```
        ┌──────────┐
        │ 讀信      │
        └────┬─────┘
             ▼
        ┌──────────┐
        │ 分類意圖  │────────┐──────────┐
        └────┬─────┘        │          │
             ▼               ▼          ▼
        ┌──────────┐   ┌──────────┐ ┌──────────┐
        │ 查文件    │   │ 建 bug 單 │ │ 真人審核  │
        └────┬─────┘   └────┬─────┘ └────┬─────┘
             └──────┬───────┴───────────┘
                    ▼
              ┌──────────┐
              │ 草擬回覆  │───────┐
              └────┬─────┘       ▼
                   ▼        ┌──────────┐
              ┌──────────┐  │ 真人審核  │
              │ 送出回覆  │  └──────────┘
              └──────────┘
```

注意：圖裡的箭頭代表「可能的路徑」，但「實際走哪一條」的決定，發生在節點裡（例如「分類意圖」這個節點決定接下來去查文件、建 bug 單、還是直接給真人）。有些節點會做決策（分類、草擬回覆、真人審核），有些則永遠走向同一個下一步（讀信永遠接分類）。

> 🩸 **血淚教訓** —— 我看過團隊把 classify、retrieve、draft 全塞進一個叫 `process_email` 的大節點，覺得「乾淨」。後來想加 streaming（看不到中間步驟）、想對 retrieve 單獨重試（包不出來）、想替 draft 加 HITL 審核（會卡死整個節點），通通做不到——只能整段拆開重寫。**LangGraph 給你的好處（streaming、retry、條件分支、HITL）全都長在「節點邊界」上**；節點切得粗看起來簡潔，其實是把未來自己想用的功能全鎖死。一個節點只做一件事，不是潔癖，是工程紀律。

**第二步：想清楚每個步驟要做什麼、需要什麼 context**

對每個節點，判斷它屬於哪一類操作、需要哪些資訊：

- **LLM 步驟**：需要理解、分析、生成、做推理決策時用。例如「分類意圖」「草擬回覆」。
- **資料步驟**：需要從外部來源取資訊時用。例如「查文件」「查客戶歷史」。可以考慮重試與快取。
- **動作步驟**：需要對外執行動作時用。例如「送出回覆」「建 bug 單」。通常要重試，且不該快取（每次送信都是獨立動作）。
- **使用者輸入步驟**：需要真人介入時用。例如「真人審核」。

每個節點還要分清楚它的兩種 context：**靜態 context**（寫在 prompt 裡的，例如分類類別定義、公司語氣準則）與**動態 context**（從 state 來的，例如信件內容、分類結果）。這個區分在第二步就想清楚，第三步設計 state 才不會亂。

**第三步：設計你的 state**

State 是所有節點共享的記憶——把它想成 agent 在處理過程中用來記錄「學到什麼、決定什麼」的筆記本。對每一份資料問兩個問題：

- **它需要跨步驟保留嗎？** 需要，就放進 state。（例如原始信件、分類結果、查到的資料、草稿。）
- **它能從別的資料推導出來嗎？** 能，就別存，用到時即時算。

以 email agent 為例，要放進 state 的有：原始信件與寄件者（事後重建不出來）、分類結果（多個後續節點要用）、檢索結果與客戶資料（重新撈很貴）、草稿（要撐過審核）、執行 metadata（除錯與回復用）。

> ⏱ **三秒判斷** —— 這份資料該放 state 嗎？
> - 需要**跨節點傳遞**或**之後再讀**？ → **放 state**
> - 能從**別的 state 欄位推導**？ → **別存**，用到再算（記憶體愈少 checkpoint 愈小、debug 愈清楚）
> - 是**組好的 prompt 文字**？ → **別存**，state 放原料、prompt 在節點裡即時組（呼應 4.4 原則）

**第四步、第五步：定義邊與編譯執行**

把節點之間「固定的轉移」與「條件分支」連起來（邊），最後編譯成可執行的圖。這兩步偏實作，是 Ch5 的主題。

## 4.4 一條關鍵原則：state 存原始資料，prompt 在 node 裡組裝

第三步藏著一個會大幅影響你程式品質的設計原則，值得獨立講：**你的 state 應該存「原始資料」，而不是「組好的文字」。需要 prompt 時，在節點內部即時組裝。**

舉例：別在 state 裡存一段「親愛的客戶您好，關於您的問題……」的成品文字；而是存「分類結果、檢索到的文件、客戶歷史」這些原始素材，等「草擬回覆」節點要跑時，才把它們組裝成 prompt。

這樣做有四個好處：

- 不同節點可以用**不同方式**格式化同一份資料（分類節點和回覆節點對「信件內容」的用法不同）。
- 你可以改 prompt 模板而**不必動 state schema**。
- **除錯更清楚**——你能精確看到每個節點實際收到什麼原始資料。
- agent 能**演化而不破壞**既有的 state。

這條原則之所以現在就講，是因為它會貫穿你之後寫的每一個 LangGraph 應用。把資料和它的呈現（presentation）分開，是好設計的通則，在 agent 裡尤其重要。

## 常見坑（Pitfalls）

- **節點切得太粗**：把「分類 + 檢索 + 草稿」塞進一個大節點，會失去 LangGraph 的好處（可觀測、可重試、可條件分支）。讓每個節點只做一件事。
- **把該算的東西也塞進 state**：state 愈肥，序列化、除錯、persistence 成本愈高。能推導的就別存。
- **在 state 裡存組好的文字而非原始資料**：之後想改 prompt、想換格式時你會很痛。存素材，臨用再組。
- **混淆「邊畫出所有可能路徑」與「實際走哪條」**：圖只是描述可能性，實際路徑由節點內的邏輯（與條件邊）在執行時決定。

## 本章小結

LangGraph 把 agent 與工作流建模成一張圖，只有三個元件：**state**（共享的狀態快照、所有節點共讀共寫的白板）、**node**（做事的函數，可含 LLM 或純程式）、**edge**（決定下一步往哪走，可固定可條件）。一句口訣：節點做事，邊決定走向。面對一個真實流程（本章用客服 email agent），用五步思考法拆解：畫出離散步驟（節點）、想清每步要做什麼與需要哪些 context、設計 state、定義邊、編譯執行。其中最該內化的設計原則是：**state 存原始資料，prompt 在節點裡即時組裝**——把資料與呈現分開，你的圖才好改、好除錯、好演化。下一章，我們就把這張 email agent 的圖，用 `StateGraph` 真正寫出來、跑起來。

## 延伸閱讀 / 練習

1. **拆你自己的流程**：挑一個你想自動化的真實流程，照 4.3 的第一步畫成節點與邊的圖。標出哪些節點會「做決策」、哪些「永遠走同一步」。
2. **分類節點**：對 email agent 的「分類意圖」節點，寫出它的靜態 context（要放 prompt）與動態 context（從 state 來）分別有哪些。
3. **state 清單**：列出 email agent 的 state 該有哪些欄位，並對每一個說明「為什麼要存、不能推導」。
4. **原則應用**：找一個你看過、把「組好的字串」存進狀態的例子（你自己的舊程式也行），說明若改成「存原始資料、臨用再組」會有什麼好處。
